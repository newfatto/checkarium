# Checkarium

Checkarium — это сайт для владельцев рептилий, амфибий, пауков и других обитателей террариумов.
Приложение помогает вести карточки питомцев, журнал событий ухода, следить за ограничениями по контакту (после кормления/линьки) и получать ежедневные Telegram-напоминания.

## Что это за проект

- Формат: веб-сайт на Django (CRUD, SSR-шаблоны, HTML/CSS, серверный рендеринг страниц).
- Назначение: персональный учёт питомцев + планирование регулярного ухода.
- Доменные сущности: пользователи, питомцы, события ухода.
- Интеграции: Telegram Bot API для привязки аккаунта и отправки уведомлений.

## Разделы и страницы сайта

Публичные/гостевые:

- `/` — главная страница с описанием сервиса и публичными карточками питомцев.
- `/users/register/` — регистрация.
- `/users/login/` — авторизация.

Для авторизованного пользователя:

- `/users/profile/<id>/` — профиль пользователя.
- `/users/profile/<id>/update/` — редактирование профиля.
- `/users/profile/<id>/delete/` — удаление профиля.
- `/pets/` — список питомцев.
- `/pets/create/` — создание питомца.
- `/pets/<id>/` — детальная карточка питомца.
- `/pets/events/` — список событий ухода.
- `/pets/events/create/<event_type>/` — создание события нужного типа.
- `/pets/events/<id>/` — детальная страница события.

Также доступны:

- `/admin/` — админ-панель Django.

## Регистрация, авторизация и роли

### Регистрация и вход

- Регистрация через `CustomUserCreationForm` (email, имя, часовой пояс, пароль).
- Авторизация через `CustomAuthenticationForm` по email + пароль.
- Модель пользователя кастомная (`CustomUser`) с `USERNAME_FIELD = email`.

### Группы пользователей и права

В проекте фактически работают 3 уровня доступа:

1. **Гость (неавторизованный)**
   - Видит главную страницу и публичных питомцев (`is_public=True`).
   - Не имеет доступа к CRUD питомцев/событий и профилю.

2. **Обычный авторизованный пользователь (владелец)**
   - Управляет только своими питомцами и своими событиями.
   - Профиль может смотреть/редактировать/удалять только свой.
   - Для событий в формах доступен выбор только своих питомцев.

3. **Модератор** (группа `Moderators`) и/или **superuser**
   - Видит всех питомцев и все события.
   - Может редактировать и удалять питомцев/события не только свои.
   - При создании события может выбирать любого питомца; владельцем события становится владелец выбранного питомца.

## Какие данные хранятся

### Пользователь (`CustomUser`)

Основные атрибуты:

- email (уникальный, логин), имя, фамилия, город, телефон;
- аватар, bio;
- Telegram-поля: `telegram_id`, токен привязки и его дата, дата привязки;
- настройки уведомлений: `care_notifications_enabled`, `last_care_notification_date`;
- `time_zone` пользователя.

### Питомец (`Pet`)

- владелец (`owner`), имя;
- тип животного (snake/lizard/turtle/frog/spider/scorpion/other);
- вид, морфа/окрас, пол;
- даты: рождения и начала владения;
- параметры: вес (г), длина (см);
- фото, заметки по кормлению, общий комментарий;
- статус (active/archived/deceased/rehomed);
- флаг публичности `is_public`;
- служебные даты создания/обновления.

### Событие (`Event`)

- владелец (`owner`) и ссылка на питомца (`pet`);
- тип события: кормление, измерение, уборка, линька, пользовательское;
- дата/время события;
- название (только для custom-события), комментарий;
- параметры повторения: `repeat_after_days`;
- ограничение на контакт: `no_handling_days`;
- измерения в событии: `weight_grams`, `length_cm`;
- дата создания.

## Валидация и бизнес-правила

### На уровне пользователей

- email валидируется и нормализуется (lowercase), проверяется уникальность без учёта регистра;
- имя/фамилия: только буквы, пробел и дефис;
- телефон: допустимый формат (регулярное выражение);
- `time_zone`: поддержка как `UTC±HH:00`, так и IANA-зон;
- изображения (аватар): только jpg/jpeg/png/webp и ограничение размера;
- нельзя включить Telegram-уведомления без привязанного `telegram_id`.

### На уровне питомцев

- имя, вид и морфа очищаются от лишних пробелов;
- пустые строки в имени/виде запрещены;
- дата рождения и дата приобретения не могут быть в будущем;
- дата приобретения не может быть раньше даты рождения;
- вес/длина валидируются диапазонами;
- фото валидируется по типу/размеру.

### На уровне событий

- событие не может быть датировано будущим временем;
- custom-событие обязано иметь `title`; другие типы не должны иметь `title`;
- для измерения нужно указать хотя бы вес или длину;
- вес/длина запрещены для событий, кроме "Измерение";
- для события "Линька" повторение (`repeat_after_days`) запрещено;
- событие можно создать только для питомца, принадлежащего указанному владельцу;
- при сохранении события "Измерение" автоматически обновляются вес/длина в карточке питомца.

## Фильтрация, сортировка и ограничения доступа

- Список событий (`/pets/events/`) поддерживает фильтрацию:
  - по питомцу (`pet`),
  - по типу события (`event_type`).
- Сортировка событий:
  - по `event_datetime` (прямой/обратный порядок).
- Ограничения выборок:
  - обычный пользователь видит только свои события/питомцев,
  - модератор и superuser — все.
- Детальная карточка питомца учитывает приватность:
  - `is_public=False` доступен только владельцу/модератору.

## Telegram: как работает и что приходит в уведомлении

### Что реализовано

1. Привязка Telegram-аккаунта через deep link `/start <token>`.
2. Запуск бота отдельной management-командой (`run_telegram_bot`, long polling).
3. Ежедневная Celery-задача для отправки напоминаний в локальное время пользователя.
4. Отправка сообщений через Telegram Bot API (`sendMessage`, HTML parse mode).

### Состав ежедневного Telegram-уведомления

Утреннее сообщение содержит:

- приветствие с именем пользователя;
- текущую дату в часовом поясе пользователя;
- блоки по каждому питомцу:
  - эмодзи по типу животного;
  - имя питомца;
  - статус контакта: можно брать на руки или лучше не трогать;
  - если есть активная линька — дата/время «до когда»;
  - список задач на сегодня:
    - «поменяй воду» (всегда),
    - повторяющиеся задачи (кормление/уборка/измерение),
    - пользовательские (custom) задачи, если их дата повтора наступила сегодня;
- ссылку на раздел событий сайта для быстрого добавления ухода.

## Админка и зарегистрированные формы

В Django admin зарегистрированы:

- `CustomUser` (кастомный `UserAdmin` c полями профиля, Telegram и правами доступа);
- `Pet` (`PetAdmin` с фильтрами/поиском/fieldsets);
- `Event` (`EventAdmin` с фильтрами/поиском/fieldsets).

То есть в админке доступны формы создания/редактирования пользователей, питомцев и событий с настроенными полями, фильтрами и поиском.

## Стек и технологии

### Backend

- Python 3.10+ (проект ориентирован на 3.13)
- Django 5.2 (CBV, ORM, Django Admin)
- Celery + Redis
- django-celery-beat (расписание периодических задач)
- Requests (интеграция с Telegram Bot API)

### Frontend

- Django Templates (SSR)
- HTML5
- CSS (статические стили в `static/css/base.css`)
- Bootstrap-классы в шаблонах

### Инфраструктура
- Docker / Docker Compose (dev + prod)

### API и интеграции

- Внешний API: Telegram Bot API (`getUpdates`, `sendMessage`).
- Внутренний HTTP-интерфейс: маршруты Django (пользователи, питомцы, события, админка).
- Дополнительный асинхронный контур: Celery worker/beat для плановых уведомлений.

## Быстрый запуск (локально)

1. Установить зависимости:
```bash
poetry install
```

2. Создать `.env` из `.env.sample` и включить `DEBUG=True` для локальной разработки. 
3. Применить миграции:
```bash
python manage.py migrate
```
4. Запустить сервер:
```bash
python manage.py runserver
```
Сайт: `http://127.0.0.1:8000/`  
Админка: `http://127.0.0.1:8000/admin/`

## Docker

Dev:
```bash
docker compose up --build
```

Prod:
```bash
docker compose -f docker-compose.prod.yml up -d --build
```
